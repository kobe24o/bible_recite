# 离线跨平台圣经背诵 App 设计规格

- 日期：2026-07-12
- 状态：设计已确认，实施计划已编写
- 首发平台：Android、iOS、Windows、macOS
- 后续平台：HarmonyOS NEXT
- 产品原则：离线优先、隐私优先、经文来源可替换、判定结果可解释

## 1. 背景与目标

本项目要开发一款帮助用户背诵基督教圣经的跨平台应用。用户安装应用后，无需联网即可浏览经文、录音背诵、完成语音识别、检查错漏、执行背诵计划、查看统计并备份个人数据。

首版的核心目标是：

1. 提供新教常用的 66 卷圣经，并支持简体中文、繁体中文和英文。
2. 通过多级目录让用户按“译本 → 新旧约 → 卷 → 章 → 节”选择背诵范围。
3. 在设备本地完成普通话和英语语音识别，不调用云端语音或 AI 服务。
4. 支持逐节背诵与连续背诵，并区分真正错误、口头语、自我纠正和识别不确定。
5. 记录背诵历史、掌握度和复习状态。
6. 支持自动与手动计划，例如“30 天背诵腓立比书”，并处理休息日、漏做、补课和顺延。
7. 所有个人数据默认只保存在本机，可通过用户密码加密的文件导出和恢复。
8. 经文来源和识别模型均通过稳定接口隔离，后续替换时不重写业务逻辑。

## 3. 首版范围与非目标

### 3.1 首版包含

- 三个离线经文包：CUV 简体、CUV 繁体、英文 WEB。
- 中英文单译本阅读和双语对照。
- 66 卷多级目录与经文范围选择。
- 今日任务、圣经、计划、统计四个一级入口。
- 逐节背诵、连续多节背诵和整章挑战。
- 离线录音、VAD 切分、普通话/英语识别和经文对齐。
- 提示首字、提示下一词、暂停、重练错句和完整重背。
- 正确、遗漏、替换、插入、乱序、自我纠正和识别不确定的解释性反馈。
- 自动计划、手动调整、学习日、休息日、缓冲日、补课和本地通知。
- 背诵历史、经节掌握度、连续天数、复习队列和趋势统计。
- 密码加密的本地备份导出/导入。
- 四个平台的持续集成构建与基础安装包。
- 经文、模型和开源依赖的来源及许可页面。

### 3.2 首版不包含

- 账号、云同步、云备份或跨设备自动同步。
- 社交、教会小组、排行榜或共享计划。
- 在线 AI、云端语音识别或服务器端判定。
- 音频圣经、讲章、注释、词典或原文工具。
- Web 客户端和 HarmonyOS NEXT 客户端。
- 应用商店内购、广告或订阅。
- 面向普通用户的任意格式经文文件导入；首版只实现稳定经文包格式和内部适配器。

## 4. 技术路线

### 4.1 客户端

Flutter 负责四个平台的界面、导航、状态管理和业务逻辑。移动端使用底部导航；桌面端在宽屏下改为左侧导航栏和双栏内容区。领域层不引用 Flutter 控件，保证判定、计划和存储逻辑可以独立测试。

推荐的代码分层为：

1. `presentation`：页面、组件、导航、响应式布局和可访问性。
2. `application`：用例编排，如开始背诵、提交片段、生成计划、完成任务和恢复备份。
3. `domain`：经文引用、背诵判定、计划规则、掌握度和接口契约。
4. `infrastructure`：SQLite、经文包、sherpa-onnx、麦克风、通知、文件和加密实现。

### 4.2 离线识别

识别层使用 sherpa-onnx 的 Dart/Flutter 接口，首版固定 SenseVoice INT8 模型。该模型文件约 228 MB，支持普通话和英语。不同平台还会包含各自的 ONNX Runtime 与原生库，因此最终安装包目标范围为约 280–400 MB。

模型必须作为安装包的一部分交付。应用不得以“识别失败”为理由静默调用任何网络服务。模型版本、文件哈希、来源和许可随构建固定并在应用中展示。

音频数据流为：

`麦克风 PCM → VAD 语音片段 → 本地 ASR → 文本标准化 → 经文序列对齐 → 结果解释 → 进度更新`

识别运行在后台 isolate/原生线程，界面线程只接收状态和结果。连续模式按 VAD 片段增量处理，最终以整段单调序列对齐为准，不能仅依靠停顿判断经节边界。

### 4.3 本地存储

只读 `scripture.sqlite` 经文包保存经文索引和并行映射；独立的 Drift `user.sqlite` 保存计划、任务、背诵尝试、掌握度、提醒登记和设置。两类数据库物理分离，便于更换经文包、制作一致备份并避免用户库迁移改动权威原文。

数据库迁移必须具备版本号、事务和回滚保护。任何迁移失败都应保留旧数据库，不得用空库覆盖。

## 5. 经文来源、格式与兼容接口

### 5.1 权威来源

首发经文从 eBible.org 的开发者格式下载：

- 新标点和合本简体：<https://ebible.org/details.php?id=cmn-cu89s>
- 新标点和合本繁体：<https://ebible.org/details.php?id=cmn-cu89t>
- World English Bible：<https://ebible.org/details.php?id=eng-web>

上述页面当前将这些版本标记为 Public Domain。构建系统仍需保存下载时的许可文本、来源 URL、内容版本日期和 SHA-256；不能只依赖本规格中的描述。

经文只由显式维护命令下载。维护者核对来源、许可、结构和哈希后，生成确定性的内部经文包，并把经文包与许可文件纳入版本控制或固定版本的项目发布资产。普通 CI 使用已经核准的经文包，不依赖 eBible.org 当时是否可访问；只有明确执行“更新经文源”流程时才重新下载并产生可审查的差异。已发布应用在运行时直接读取内置经文包。

eBible VPL ZIP 同时包含显示文本和 SQL 元数据。显示文本中的合并节只给出起始节号，必须从伴随 SQL 的 `startVerse/endVerse` 恢复完整范围；构建工具只解析受限 `INSERT` 字段，绝不能执行上游 SQL。原文 Unicode、显式空节和合并节范围均原样保留。

### 5.2 稳定经文定位

所有业务记录使用稳定引用键：

`VerseKey(canonId, osisBookId, chapter, verse)`

- `canonId` 首版固定为 Protestant 66-book canon，但字段不可省略。
- `osisBookId` 使用稳定的 OSIS 经卷标识，不使用界面显示名作为主键。
- `VerseKey` 提供跨译本共同位置；具体尝试、计划和掌握度同时记录译本 ID、包 ID、versification ID 和语义内容 SHA-256。不同译本或同名译本的不同文本修订分别掌握，再按共同 `VerseKey` 汇总；不得让替换来源静默继承不同措辞的掌握度。
- 跨译本章节结构不一致时，经文包必须提供显式映射；不得假设所有来源完全同构。

经文存储分成两层：

- `VerseUnit` 是来源中的一个文本单元，具有起止节号；合并节的原文只保存和显示一次。
- `VerseSlot` 是可寻址的单个 `VerseKey`，多个槽位可以指向同一个合并文本单元；显式省略的节保存为 `omitted`，不能直接删除。

对固定首发快照的真实结构检查表明，CUV 简繁在展开合并节前后与 WEB 有 67 个章节的节号集合不同。利用 VPL SQL 的 70 个 CUV 合并节后，仍有 12 个章节需要人工审查映射，包括约翰三书 1:14–15、启示录 12:18/13:1 和罗马书 16:25–27/14:24–26 等多对一、跨章或移位关系。因此双语界面必须消费内容哈希绑定的显式多对多映射组；不得按列表位置或相同节号机械拼接。任一译本内容哈希变化都会使旧人工映射失效并阻止发布，直到重新审核。

### 5.3 接口边界

运行时只依赖 `ScriptureRepository`：

```text
listTranslations()
getTranslation(id)
listBooks(translationId, canonId)
getChapter(translationId, bookId, chapter)
getPassage(translationId, passageRange)
getSelection(translationId, passageSelection)
resolveParallelPassage(locatedSourceRange, targetTranslationId) -> ParallelPassage
```

`ParallelPassage` 由有序的 `ParallelGroup` 组成。每组分别包含零个或多个源 `VerseUnit`、零个或多个目标 `VerseUnit`、关系类型（同节、源合并、目标合并、跨章、移位、源缺失或目标缺失）以及自动/人工审核来源。调用参数必须包含源译本 ID，避免在不同 versification 间产生隐含假设。

构建侧使用 `ScriptureSourceAdapter` 将 USFM、VPL 或未来授权格式转换为内部经文包。应用侧使用 `ScripturePackValidator` 校验包结构、许可元数据、正典、重复经节、缺失经节和哈希。

经文包格式包含：

- `manifest.json`：包 ID、schema 版本、译本、语言、正典、versification ID、来源、版本、许可、文本单元/槽位计数、语义内容哈希和映射哈希。
- `scripture.sqlite`：规范化的卷、章、文本单元、节号槽位和经内容哈希验证的并行对照组。
- `LICENSE.txt`：可离线查看的完整许可与署名。
- 可选签名：未来从非安装包渠道导入时用于验证发布者。

## 6. 信息架构与核心流程

### 6.1 一级导航

- **今日**：当前计划、今日新内容、到期复习、连续天数和快速继续。
- **圣经**：译本切换、多级目录、范围选择、双语对照和加入计划。
- **计划**：新建、编辑、日历、补课、暂停和完成情况。
- **统计**：背诵历史、掌握度、复习队列、经卷进度和趋势。

### 6.2 选择经文

用户依次选择译本、新约/旧约、经卷、章节和节范围。可直接开始逐节或连续背诵，也可把范围加入新计划。切换译本时应尽量保留当前位置。

### 6.3 背诵

背诵开始后默认隐藏经文，避免看着念。界面显示范围、模式、录音状态、时长和离线提示。用户可以暂停、请求首字或下一词提示。

逐节模式每节独立判定，通过后进入下一节。连续模式允许一次背诵多节或整章，结束后统一对齐，并按经节展示结果。

原始录音默认只存在于当前会话的临时缓冲中，判定完成后删除。用户可主动开启“保存失败录音用于自己复听”，该选项默认关闭。

### 6.4 结果

结果页使用文字、图标和颜色共同表达：

- 正确内容。
- 已完成的自我纠正。
- 未修正的遗漏、替换或乱序。
- 模型不确定、需要用户重说的片段。

用户可以只重练错句、完整重背或继续下一段。结果写入本地后更新计划任务、经节掌握度和复习队列。

## 7. 背诵判定规则

### 7.1 原文与比较副本

应用必须保存并显示经文原文，不得为了匹配而改写。判定只使用临时比较副本：

- 中文忽略标点、空白和显示用简繁差异，并统一常见数字读法。
- 英文忽略标点、大小写和不影响内容的缩写形式差异。
- 译本中的实词、专名和语序仍需匹配，不能用“语义相近”自动放行。
- 章节、节号和用户主动说出的引用提示不参与正文评分。

### 7.2 序列对齐

判定器对目标经文与识别文本做单调序列对齐，产生：正确、插入、删除、替换和乱序操作。中文结合字符与词边界，英文按单词和规范化词形处理。

连续模式必须把整个识别序列与整个目标范围对齐，再投影回各节。用户停顿位置不能改变经文顺序或制造虚假的经节边界。

### 7.3 容错规则

- 可配置的无意义口头语，如“嗯”“那个”，在不改变经文内容时忽略。
- 重复正确短语不直接判错，但会影响流畅度。
- 用户说错后在局部窗口内完整重说正确内容，前一错误标记为“已自我纠正”。
- 自我纠正不阻止内容通过，但会降低熟练度。
- 经文专名、同音字或低可信片段先走本地发音辅助匹配；仍无法判断时标记“请重说”，不得直接判错或自动判对。
- 请求提示会降低熟练度；显示完整经文会结束当前无提示尝试。

### 7.4 通过状态与熟练度

内容通过条件：完成不确定项重说后，不存在未修正的遗漏、替换或乱序。口头语、纯重复和已完成的自我纠正不导致内容失败。

结果分为：

1. **准确通过**：内容通过，且没有提示和明显自我纠正。
2. **通过但需复习**：内容通过，但提示、自我纠正或超时较多。
3. **需要重练**：仍有未修正的内容错误。

熟练度为 0–100 的独立指标，综合提示、自我纠正、重复、用时和近期稳定性。熟练度影响复习日期，但不能覆盖内容正确性。

## 8. 计划与复习

### 8.1 自动计划输入

用户选择经文范围、开始日期、截止日期、每周学习日、休息日和可选缓冲日。计划生成器按照经文字数、节数和历史难度平衡每日工作量，不简单平均章节数量。

每天可以同时包含：

- 新背内容。
- 到期复习。
- 以前未通过的重练内容。

### 8.2 手动调整

用户可以移动、拆分、合并或锁定某日任务。自动重排不得覆盖锁定项。用户也可以创建完全手工的计划。

### 8.3 漏做与补课

漏做任务不得被自动标记完成。重排顺序为：

1. 优先使用尚未占用的缓冲日。
2. 在不修改锁定项和休息日的前提下，将剩余工作量均摊到未来学习日。
3. 若截止日期已无法满足，明确提示每日新增负担，并让用户选择延长日期或接受更高负担。

### 8.4 复习间隔

首次通过后采用 1、3、7、14、30 天的基础复习阶梯。准确且稳定通过可进入下一间隔；提示较多或失败会停留或缩短间隔。用户可手动提前复习，但不得因此自动跳过计划中的正式任务。

提醒使用系统本地通知，无需服务器。

## 9. 数据模型

核心实体如下：

| 实体 | 主要职责 |
|---|---|
| `ScripturePack` | 来源、译本、语言、正典、版本、许可、schema 和校验值 |
| `BibleBook` / `VerseUnit` / `VerseSlot` | 稳定经卷标识、来源文本单元、合并节范围、可寻址节号和省略状态 |
| `ParallelGroup` | 两个 versification 间经审核的多对多文本单元关系及来源证据 |
| `PassageRange` | 同一正典、同一卷内的一个连续起止范围 |
| `PassageSelection` | 一个或多个有序、不重叠 `PassageRange`，表达连续或离散选择 |
| `Plan` | 目标、日期、学习日、休息日、缓冲和状态 |
| `PlanTask` | 某日的新背、复习或补课任务及锁定状态 |
| `PracticeSession` | 一次正在进行的背诵会话和临时状态 |
| `PracticeAttempt` | 范围、经文包修订、本地日期/UTC 时间、错漏、提示、用时、通过状态和熟练度 |
| `VerseMastery` | 每个经节与语义内容修订的掌握度、复习间隔和下次复习日期 |
| `ReviewQueueItem` | 到期复习、失败重练和优先级 |
| `AppSettings` | 界面语言、默认译本、模型、通知、隐私和可访问性设置 |
| `BackupManifest` | 备份 schema、创建时间、加密参数、内容摘要和完整性校验 |

用户数据引用经文稳定键与内容修订身份，不能使用数据库自增 ID 作为跨备份或跨经文包的唯一身份。背诵发生时同时保存本地日历日期和时区信息，之后旅行或修改时区不能重写历史连续天数。

## 10. 备份、安全与隐私

### 10.1 本地默认

- 应用不需要账号。
- 应用运行时没有业务网络依赖。
- Android 构建不声明非必要的互联网权限；其他平台不实现后台网络客户端。
- 原始录音默认不保存；用户主动保留时，以 SHA-256 内容寻址的不可变本地文件保存。
- 统计只在本机生成，不接入远程分析 SDK。

### 10.2 加密备份

备份包含用户数据库、设置和可选的用户主动保存录音，不重复打包可从安装包恢复的模型与首发经文。

备份使用版本化加密信封：Argon2id 从用户密码派生密钥，XChaCha20-Poly1305 提供加密与完整性保护。盐、算法参数、schema 版本、精确密文长度和所需经文包修订写入经认证的 `BackupManifest`；Poly1305 tag 是密文完整性的权威校验，不增加未认证摘要。密码不保存，丢失后无法恢复。

导入时先在临时位置完成密码验证、完整性校验、schema 迁移和引用检查。首版只支持“完整恢复”：使用 SQLite 一致快照、经 fsync 的恢复日志和平台原子替换，不复制活动 WAL 主文件。替换提交点前失败继续使用旧数据；提交点后若进程中断，下一次启动完成新库的提醒与录音后置工作。缺少完全相同的经文包修订时仍可恢复历史，但标为“经文包暂不可用”，不能绑定到同名不同内容。用户主动包含的录音先按内容哈希安装，再提交引用它们的数据库；不实现自动合并。

## 11. 异常处理

| 场景 | 行为 |
|---|---|
| 麦克风权限被拒绝 | 阅读、计划和统计继续可用；背诵页说明原因并提供系统设置入口 |
| 模型缺失、哈希错误或加载失败 | 禁止开始录音并显示修复信息；不得切换到云端识别 |
| 设备空间不足 | 在首次准备模型和导入备份前检查空间，不进行半成品写入 |
| 没有检测到语音 | 本次不计失败，提示检查麦克风后重试 |
| 噪声过大或片段无法识别 | 标记片段需重说，不写入错误记录 |
| 电话、锁屏或切后台 | 安全暂停录音；返回后由用户继续或放弃，不擅自提交半段结果 |
| 设备过热或识别明显变慢 | 降低并行线程并提示用户；不牺牲正确性规则 |
| 经文包损坏 | 隔离损坏译本；其他译本和用户数据继续可用 |
| 数据库迁移失败 | 回滚并保留旧库，提供本地诊断信息 |
| 备份密码错误或文件损坏 | 不修改现有数据，明确区分密码错误与完整性错误 |
| 计划已无法按期完成 | 展示新增日负担，让用户选择延期或重排 |

## 12. 可访问性与本地化

- 界面至少提供简体中文、繁体中文和英文资源。
- 经文显示字体、字号、行距和主题可调整。
- 所有颜色反馈必须同时配有图标和文字。
- 录音、暂停、结束和提示按钮要支持屏幕阅读器标签和键盘操作。
- Windows/macOS 支持键盘导航和可见焦点。
- 大字号下不得截断经文、错误说明或主要操作。

## 13. 测试策略与验收门槛

### 13.1 单元测试

- 中文与英文文本标准化。
- 插入、删除、替换、乱序和重复的序列对齐。
- 自我纠正、口头语和提示规则。
- 计划拆分、休息日、缓冲日、锁定项、漏做和截止日期重排。
- 掌握度与复习间隔。
- 数据库迁移、备份加密和恢复校验。
- 经文适配器与经文包验证。

### 13.2 经文完整性测试

- 验证 66 卷及每卷章节结构。
- 对比来源文件的经节数量和规范化哈希。
- 检查重复、缺失、非法章节和非法节号。
- 检查合并节只保存一个文本单元但所有节号槽位均可定位，WEB 的显式空节保留为 `omitted`。
- 固定验证原始 67 个差异章节、合并节展开后的 12 个审核章节，以及全部正反向并行映射零 `unresolved`。
- 检查简繁及英文的同节、一对多、多对一、跨章、移位和缺失关系均由显式映射正确定位。
- 修改任一经文包语义内容哈希时，旧人工映射必须失效并阻止构建。
- 构建产物必须包含来源、许可和版本信息。

### 13.3 音频与判定测试

建立获得授权的本地测试集，覆盖普通话、英语、男女声、不同语速、停顿、轻度环境噪声、经文专名、重复和自我纠正。最小验收集包含普通话与英语各至少 8 名说话者、每种语言至少 150 条完全正确样本和 75 条预先标注的故意错漏样本；同一说话者不能同时贡献开发调参集和最终验收集。原始测试录音不得进入公开仓库，除非录音者明确同意相应许可。

首版发布门槛：

- 在验收音频集中，完整正确背诵的误判失败率不超过 3%。
- 故意遗漏关键内容的误判通过率不超过 1%。
- “不确定”必须要求重说，不能计入上述自动通过。
- 中档参考手机上模型冷启动目标不超过 8 秒。
- 持续识别处理速度必须快于实际录音时长，目标实时因子不高于 0.5。
- 连续背诵 30 分钟不得崩溃、丢失已完成结果或出现持续增长的内存占用。

### 13.4 跨平台与离线验收

Android、iOS、Windows、macOS 分别验证：

- 安装、首次启动和模型加载。
- 麦克风授权、拒绝后恢复、切后台和锁屏。
- 逐节与连续背诵的完整闭环。
- 本地通知。
- 加密备份导出、导入和错误密码。
- 不同屏幕尺寸、键盘导航和大字体。
- 飞行模式或完全阻断网络时，浏览、识别、判定、计划、统计和备份均可使用。

## 14. 构建与发布

持续集成按平台构建：

- Android：APK/AAB。
- iOS：无签名测试构建；正式 IPA 需要用户提供 Apple 签名与发布凭据。
- Windows：x64 压缩包和带包身份的 MSIX；通知发行验收以 MSIX 为准。
- macOS：分别构建 arm64/x64；正式分发需要 Apple 签名和公证凭据。

普通 CI 使用已核准且固定版本的经文包。ASR 模型只在一个可复用准备任务中从固定发布资产下载、验证 SHA-256，并与完整 FunASR 模型协议、SenseVoice 署名/模型名、Silero、sherpa-onnx、ONNX Runtime 及插件许可共同形成不可变输入产物；四平台消费同一产物。经文更新使用独立维护流程，从权威来源重新生成经文包并提交可审查的内容、版本、许可和哈希变化。不可使用未经固定版本或未经校验的“latest”经文、模型或许可进入发布包。

任何发布凭据、证书和密钥不得提交到仓库。

## 15. 成功标准

首版完成的定义是：用户可在四个平台安装应用，在完全断网的情况下选择 66 卷中的任意经文，用普通话或英语完成逐节或连续背诵，得到可解释且符合容错规则的结果；应用能据此更新本地记录、计划和复习队列，并能通过加密文件可靠备份与恢复个人数据。经文与模型的来源、版本、许可和校验信息可追溯，后续新增来源不需要重写背诵、计划或统计模块。
